HTTPS的工作原理

2019年9月13日

HTTPS 和 HTTP 协议相比提供了

  • 数据完整性：内容传输经过完整性校验
  • 数据隐私性：内容经过对称加密，每个连接生成一个唯一的加密密钥
  • 身份认证：第三方无法伪造服务端（客户端）身份

原理

HTTPS 在传输数据之前需要客户端与服务端之间进行一次握手，在握手过程中将确立双方加密传输数据的密码信息。TLS/SSL 协议是一套加密传输的协议，使用了非对称加密，对称加密以及 HASH 算法。以浏览器和服务器为例，HTTPS 通信过程如下：

第一次握手

  • 浏览器请求建立 SSL 链接，并向服务端发送一个随机数 Client random，即 Random1 和浏览器支持的加密套件。

第二次握手

  • 服务器收到请求，选择一种浏览器支持的加密方法、一个随机数 Server random，即 Random2，返回给浏览器。

  • Certificate 服务器将自己的 CA 证书下发给浏览器，让浏览器验证自己的身份，客户端验证通过后取出证书中的 public key 公钥。

第三次握手

  • Certificate Verify，浏览器收到服务器传来的证书后，先验证 CA 证书的合法性，验证通过后取出证书中的服务器 public key 公钥，再生成一个随机数 Random3，再用服务器 public key 公钥非对称加密 Random3 生成 PreMaster Key，然后把 PreMaster Key 发送服务器。浏览器也就拥有 Random1 + Random2 + Random3，根据算法就可以生成一份 Master Key。

第四次握手

服务器再用自己的私钥取出 PreMaster Key 得到浏览器生成的 Random3。服务器也拥有 Random1 + Random2 + Random3，根据浏览器同样的算法就可以生成一份 Master Key，每次传递数据都要验证 Master Key。

注意

采用 HTTPS 协议的服务器必须要有自己制作或者 CA 证书，区别就是自己颁发的证书需要客户端验证通过，才可以继续访问，而使用 CA 证书则不会弹出提示页面。这套证书其实就是一对公钥和私钥。公钥给别人加密使用，私钥给自己解密使用。

浏览器收到服务器的回复后，首先浏览器会从内置的证书列表中索引，查找服务器下发 CA 证书，如果没有找到，此时就会提示用户该证书是不可信任的。如果查到了对应 CA 证书，则取出 public key 公钥。然后，用 public key 公钥解密得到证书的内容和证书签名，内容包括证书的颁发机构，过期时间等。浏览器会先验证证书签名的合法性，签名通过后，会检查证书有效期，证书过期了也会提示用户。这些都通过认证时，浏览器就可以安全使用 public key 公钥了。

单向认证：
上图中的第二次握手，其实就是一次服务器->浏览器的单向验证，服务器将自己的 CA 证书下发给浏览器，让浏览器验证自己的身份。

双向认证：
双向认证和单向认证原理基本差不多，只是除了浏览器需要认证服务器 CA 证书以外，增加了服务器对浏览器的 CA 证书认证。
